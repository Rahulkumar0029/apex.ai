# Requirements Document

## Introduction

An AI-powered interview preparation platform that enables users to practice technical and behavioral interviews in a realistic, real-time environment. The platform simulates a live interview by generating contextually relevant questions, processing spoken answers via speech-to-text, evaluating responses using AI, and producing detailed performance reports with actionable feedback. Users can track their progress over time through analytics and history features, and manage their profile and preferences through a settings panel.

The system consists of a React 19/TypeScript frontend, a Node.js/Express/TypeScript backend, a PostgreSQL database (via Prisma), Cloudinary for media storage, Google Gemini or OpenAI for AI services, AssemblyAI or Deepgram for speech processing, and Socket.io for real-time communication.

---

## Glossary

- **Platform**: The full-stack AI-powered interview preparation system described in this document.
- **User**: An authenticated individual using the Platform to practice interviews.
- **Guest**: An unauthenticated visitor browsing public pages.
- **AI_Engine**: The backend service integrating Google Gemini or OpenAI APIs for question generation, evaluation, and feedback.
- **Speech_Service**: The backend service integrating AssemblyAI or Deepgram for real-time speech-to-text and text-to-speech.
- **Auth_Service**: The backend service responsible for registration, login, JWT issuance, refresh, and Google OAuth.
- **Interview_Service**: The backend service managing interview session lifecycle, question flow, and answer collection.
- **Report_Service**: The backend service that aggregates evaluation scores and generates interview reports.
- **Analytics_Service**: The backend service that computes aggregated performance metrics across interview history.
- **Socket_Server**: The Socket.io server managing real-time bidirectional communication during an interview session.
- **Lobby**: The pre-interview screen where the User checks camera, microphone, speaker, and internet connectivity before starting.
- **Interview_Room**: The real-time interview interface where questions are presented, answers are recorded, and AI feedback is streamed.
- **Report**: The structured post-interview document containing scores, strengths, weaknesses, suggestions, transcript, and timeline.
- **Access_Token**: A short-lived JWT used to authenticate API requests.
- **Refresh_Token**: A long-lived token stored server-side and in an HTTP-only cookie, used to issue new Access_Tokens.
- **Question**: A single interview prompt generated by the AI_Engine for a given role, experience level, difficulty, and interview type.
- **Response**: A User's recorded answer to a Question, including transcript and audio metadata.
- **Session**: A server-side record of an active or completed interview run, linked to a User and a set of Questions and Responses.
- **Plan**: A subscription tier (free or paid) that governs feature access and usage limits.
- **Activity_Log**: A server-side record of significant User actions for audit and analytics purposes.
- **Streak**: The count of consecutive days on which the User has completed at least one interview.

## Requirements

### Requirement 1: User Registration and Authentication

**User Story:** As a Guest, I want to create an account and log in securely, so that I can access my personal interview history, reports, and settings.

#### Acceptance Criteria

1. WHEN a Guest submits a registration form with a valid email, password (minimum 8 characters), and display name, THE Auth_Service SHALL create a new User record, hash the password using bcrypt, and return an Access_Token and a Refresh_Token.
2. IF a Guest submits a registration form with an email that already exists in the database, THEN THE Auth_Service SHALL return a 409 Conflict error with a descriptive message.
3. IF a Guest submits a registration form with a password shorter than 8 characters, THEN THE Auth_Service SHALL return a 422 Unprocessable Entity error listing the validation failure.
4. WHEN a User submits valid login credentials (email and password), THE Auth_Service SHALL verify the password hash, issue a new Access_Token (expiry: 15 minutes) and a new Refresh_Token (expiry: 7 days), and store the Refresh_Token server-side.
5. IF a User submits incorrect login credentials, THEN THE Auth_Service SHALL return a 401 Unauthorized error without revealing which field is incorrect.
6. WHEN a User initiates Google OAuth login, THE Auth_Service SHALL redirect to Google's authorization endpoint, exchange the authorization code for a Google ID token, and upon successful verification create or retrieve the User record and issue an Access_Token and Refresh_Token.
7. WHEN a User submits a valid Refresh_Token, THE Auth_Service SHALL invalidate the old Refresh_Token, issue a new Access_Token and a new Refresh_Token, and return both to the caller.
8. IF a User submits an expired or revoked Refresh_Token, THEN THE Auth_Service SHALL return a 401 Unauthorized error and clear the associated cookie.
9. WHEN a User logs out, THE Auth_Service SHALL revoke the active Refresh_Token server-side and instruct the client to clear the authentication cookie.
10. WHEN a User requests a password reset with a registered email, THE Auth_Service SHALL send a time-limited reset link (valid for 1 hour) to that email address.
11. WHEN a User submits a valid password reset token with a new password meeting the minimum length, THE Auth_Service SHALL update the password hash and invalidate all existing Refresh_Tokens for that User.

### Requirement 2: Interview Configuration

**User Story:** As a User, I want to configure an interview session by selecting role, experience level, difficulty, interview type, tech stack, and language, so that the questions are relevant to my target position.

#### Acceptance Criteria

1. THE Platform SHALL present the User with a multi-step interview creation form collecting: target role (free text, max 100 characters), years of experience (numeric, 0–30), difficulty level (Easy, Medium, Hard), interview type (Technical, Behavioral, Mixed), tech stack (multi-select from a predefined list, minimum 1), and preferred language (selection from supported locales).
2. IF the User submits the interview creation form with any required field missing, THEN THE Platform SHALL highlight the missing fields and prevent form submission.
3. WHEN the User completes and submits a valid interview configuration, THE Interview_Service SHALL create a new Session record with status "configured" and return the Session ID to the client.
4. THE Interview_Service SHALL accept only the following difficulty values: "Easy", "Medium", "Hard"; any other value SHALL cause a 422 error.
5. THE Interview_Service SHALL accept only the following interview type values: "Technical", "Behavioral", "Mixed"; any other value SHALL cause a 422 error.
6. WHERE a User's Plan is "free", THE Interview_Service SHALL limit the User to a maximum of 5 interview sessions per 24-hour period.

### Requirement 3: Pre-Interview Lobby and Device Check

**User Story:** As a User, I want to verify my camera, microphone, speaker, and internet connection before starting an interview, so that technical issues do not interrupt the session.

#### Acceptance Criteria

1. WHEN the User navigates to the Lobby for a configured Session, THE Platform SHALL request browser permissions for camera and microphone and display the live camera preview within 2 seconds of permission being granted.
2. IF the User denies camera or microphone permission, THEN THE Platform SHALL display a descriptive error message and disable the "Ready" button until permissions are granted.
3. WHEN the Lobby page loads, THE Platform SHALL execute an internet speed check and display the result (in Mbps) to the User.
4. IF the measured internet speed is below 1 Mbps, THEN THE Platform SHALL display a warning that the connection may be insufficient for a smooth interview experience.
5. WHEN the User clicks the speaker test button, THE Platform SHALL play a short audio sample and ask the User to confirm whether audio was audible.
6. WHEN all device checks pass and the User clicks the "Ready" button, THE Platform SHALL transition the User to the Interview_Room and emit a "Ready" event to the Socket_Server.

---

### Requirement 4: Real-Time Interview Session

**User Story:** As a User, I want to conduct an AI-driven interview in real time with spoken questions and answer recording, so that I can practice in a realistic interview environment.

#### Acceptance Criteria

1. WHEN the Interview_Room loads, THE Socket_Server SHALL emit a "startInterview" event to the client, and THE AI_Engine SHALL generate the first Question based on the Session configuration.
2. WHEN a Question is ready, THE Speech_Service SHALL synthesize the Question text to audio and THE Interview_Room SHALL play the audio to the User and display the Question text simultaneously.
3. WHILE a Question is being presented, THE Interview_Room SHALL display a countdown timer showing the remaining response time; the default response window SHALL be 120 seconds per question.
4. WHEN the User activates the microphone, THE Speech_Service SHALL stream audio to the speech-to-text engine and display a live transcript in the Interview_Room.
5. WHEN the User submits an answer (either by clicking "Submit" or when the timer expires), THE Interview_Service SHALL record the Response, including the full transcript and timestamp, and emit an "answer" event to the Socket_Server.
6. WHEN the "answer" event is received, THE AI_Engine SHALL evaluate the Response and emit a "thinking" event to the client; THE Interview_Room SHALL display the AI state "Analyzing answer..." during this period.
7. WHEN the AI_Engine has completed evaluation, THE AI_Engine SHALL generate a follow-up or next Question and emit a "nextQuestion" event; THE Interview_Room SHALL transition to the next Question.
8. WHEN the AI_Engine is generating a follow-up Question, THE Interview_Room SHALL display the AI state "Finding follow-up question...".
9. WHEN the AI_Engine is generating feedback after all questions are complete, THE Interview_Room SHALL display the AI state "Generating feedback...".
10. WHEN the User clicks "End Interview" or all Questions have been answered, THE Interview_Service SHALL set the Session status to "completed" and emit a "report" event to the Socket_Server.
11. IF the WebSocket connection is lost during an active Session, THEN THE Socket_Server SHALL attempt to reconnect the client for up to 30 seconds before marking the Session as "interrupted".
12. WHILE the User is in the Interview_Room, THE Interview_Room SHALL display a mute/unmute toggle, a leave button (which prompts for confirmation), and a camera on/off toggle.
13. THE Interview_Service SHALL support a configurable number of questions per session (minimum 3, maximum 20); the default SHALL be 8.

---

### Requirement 5: AI Question Generation and Evaluation

**User Story:** As a User, I want the AI to generate relevant questions and evaluate my answers accurately, so that I receive meaningful, role-specific feedback.

#### Acceptance Criteria

1. WHEN the AI_Engine generates a Question, THE AI_Engine SHALL produce a Question relevant to the configured role, experience level, difficulty, interview type, and tech stack.
2. WHEN the AI_Engine generates a follow-up Question, THE AI_Engine SHALL take into account the User's previous Response transcript to produce a contextually relevant follow-up.
3. WHEN the AI_Engine evaluates a Response, THE AI_Engine SHALL produce a score (0–100) for each of: technical accuracy, communication clarity, problem-solving approach, and grammar/fluency.
4. WHEN the AI_Engine evaluates a Response, THE AI_Engine SHALL produce a list of at least one strength and at least one improvement suggestion specific to that Response.
5. IF the Response transcript is empty (no speech detected), THEN THE AI_Engine SHALL assign a score of 0 for all dimensions and note "No response provided" in the feedback.
6. THE AI_Engine SHALL complete Question generation within 5 seconds and Response evaluation within 10 seconds under normal load conditions.
7. IF the AI_Engine external API returns an error, THEN THE Interview_Service SHALL retry the request up to 3 times with exponential backoff before returning a 503 Service Unavailable error to the client.

---

### Requirement 6: Speech-to-Text and Text-to-Speech

**User Story:** As a User, I want my spoken answers transcribed in real time and AI questions read aloud, so that the interview feels natural and accessible.

#### Acceptance Criteria

1. WHEN the User activates the microphone in the Interview_Room, THE Speech_Service SHALL begin streaming audio to the speech-to-text provider and return incremental transcript updates to the client within 500 milliseconds of speech detection.
2. WHEN the User deactivates the microphone or submits an answer, THE Speech_Service SHALL finalize the transcript and store it in the Response record.
3. THE Speech_Service SHALL support transcription in all languages available in the configured speech-to-text provider; the Platform SHALL present only supported locales in the language selector.
4. WHEN the AI_Engine produces a Question text, THE Speech_Service SHALL synthesize it to audio and return the audio stream to the Interview_Room for playback.
5. IF the speech-to-text provider returns a low-confidence transcript (confidence score below 0.6), THEN THE Platform SHALL display an indicator prompting the User to repeat or clarify the answer.
6. THE Speech_Service SHALL parse incoming audio streams and serialize transcription results to JSON; FOR ALL valid transcription inputs, parsing the JSON result and re-serializing it SHALL produce an equivalent object (round-trip property).

### Requirement 7: Interview Report Generation

**User Story:** As a User, I want a detailed report after each interview, so that I can understand my performance and identify areas for improvement.

#### Acceptance Criteria

1. WHEN a Session status is set to "completed", THE Report_Service SHALL generate a Report containing: overall score (weighted average of all Response scores), technical score, communication score, confidence score, grammar/fluency score, problem-solving score, aggregated strengths list, aggregated weaknesses list, improvement suggestions, question-and-answer timeline, and full transcript.
2. THE Report_Service SHALL make the Report available via GET /report/:id within 30 seconds of the Session being marked "completed".
3. WHEN a User requests a Report for a Session that belongs to a different User, THE Report_Service SHALL return a 403 Forbidden error.
4. WHEN a User requests a PDF export of a Report, THE Report_Service SHALL generate and return a downloadable PDF file containing all Report sections within 10 seconds.
5. WHEN a User clicks "Share Report", THE Platform SHALL generate a unique, time-limited share URL (valid for 7 days) that allows read-only access to the Report without authentication.
6. THE Report_Service SHALL calculate the overall score as the arithmetic mean of all per-Response overall scores, rounded to the nearest integer.
7. IF a Session has zero completed Responses at the time of Report generation, THEN THE Report_Service SHALL return a Report with all scores set to 0 and a note indicating "No responses recorded".

### Requirement 8: Dashboard

**User Story:** As a User, I want a dashboard showing my performance summary and quick actions, so that I can monitor my progress and start a new interview easily.

#### Acceptance Criteria

1. WHEN a User navigates to the Dashboard, THE Platform SHALL display: total interviews completed, average overall score across all completed Sessions, current Streak (consecutive days with at least one completed interview), and a weekly progress chart.
2. WHEN a User navigates to the Dashboard, THE Platform SHALL display the 5 most recent completed Sessions with their date, role, difficulty, and overall score.
3. WHEN a User navigates to the Dashboard, THE Platform SHALL display AI-generated suggestions (minimum 1, maximum 3) personalized based on the User's recent performance patterns.
4. THE Platform SHALL display a "Quick Start" button that navigates the User directly to the interview creation form.
5. WHEN the User's Streak reaches 7 days, THE Platform SHALL display a congratulatory notification.
6. THE Analytics_Service SHALL recompute dashboard statistics within 5 minutes of a new Session being marked "completed".

---

### Requirement 9: Analytics

**User Story:** As a User, I want detailed analytics visualizations of my interview performance over time, so that I can identify trends and focus my preparation.

#### Acceptance Criteria

1. WHEN a User navigates to the Analytics page, THE Analytics_Service SHALL return aggregated performance data grouped by: weekly and monthly time periods, target role, and difficulty level.
2. THE Platform SHALL render a line chart showing the User's overall score trend across all completed Sessions, ordered chronologically.
3. THE Platform SHALL render a radar chart showing the User's average scores across the 5 dimensions: technical accuracy, communication, confidence, grammar/fluency, and problem-solving.
4. THE Platform SHALL render a pie chart showing the distribution of completed Sessions by interview type (Technical, Behavioral, Mixed).
5. WHEN the User selects a date range filter, THE Analytics_Service SHALL return data scoped to that range within 3 seconds.
6. THE Analytics_Service SHALL include the total count of completed Sessions and the count of unique roles practiced in the analytics response.

---

### Requirement 10: Interview History

**User Story:** As a User, I want to browse, search, and manage my past interviews, so that I can review performance and keep my history organized.

#### Acceptance Criteria

1. WHEN a User navigates to the History page, THE Platform SHALL display all completed and interrupted Sessions for that User, paginated at 20 items per page, ordered by date descending.
2. WHEN a User enters a search query, THE Platform SHALL filter the Session list by role name (case-insensitive substring match) and return results within 2 seconds.
3. WHEN a User applies filters (by difficulty, interview type, date range, or score range), THE Platform SHALL return only Sessions matching all selected filter criteria.
4. WHEN a User clicks "Download Report" on a Session, THE Platform SHALL initiate the PDF generation flow described in Requirement 7, Criterion 4.
5. WHEN a User clicks "Delete" on a Session and confirms the deletion prompt, THE Platform SHALL permanently delete the Session, its associated Questions, Responses, and Report from the database.
6. IF a User attempts to delete a Session belonging to another User, THEN THE Platform SHALL return a 403 Forbidden error.
7. WHERE the Platform supports interview replay in a future release, THE Platform SHALL store the full audio recording and transcript to enable playback.

---

### Requirement 11: User Profile and Settings

**User Story:** As a User, I want to manage my profile information and application preferences, so that the platform reflects my identity and preferred experience.

#### Acceptance Criteria

1. WHEN a User updates their profile (display name, college, years of experience, skills list, or resume file), THE Platform SHALL validate and persist the changes and return a 200 response with the updated User record.
2. WHEN a User uploads a profile photo, THE Platform SHALL upload the image to Cloudinary, store the resulting URL in the User record, and display the new photo within 5 seconds of upload completion.
3. IF a User uploads a profile photo larger than 5 MB, THEN THE Platform SHALL reject the upload with a 413 error and display a descriptive message.
4. WHEN a User links or unlinks a Google account, THE Auth_Service SHALL update the User's linked accounts without requiring re-registration.
5. WHEN a User toggles the theme preference (light/dark/system), THE Platform SHALL apply the selected theme immediately without a full page reload.
6. WHEN a User updates notification preferences, THE Platform SHALL persist the preferences and apply them to subsequent notifications within the same session.
7. WHEN a User selects an AI voice preference for text-to-speech, THE Speech_Service SHALL use the selected voice for all subsequent Question audio playback in new sessions.
8. WHEN a User requests account deletion and confirms via a secondary confirmation prompt, THE Platform SHALL permanently delete all User data (profile, Sessions, Responses, Reports, Refresh_Tokens, Activity_Logs) within 30 seconds and log the User out.
9. IF a User requests account deletion but has a Session in status "active", THEN THE Platform SHALL terminate the active Session before proceeding with deletion.

### Requirement 12: Notifications

**User Story:** As a User, I want to receive notifications about interview reminders, streak milestones, and report readiness, so that I stay engaged and informed.

#### Acceptance Criteria

1. WHEN a Report is ready for a completed Session, THE Platform SHALL send an in-app notification to the User containing the Session role and a link to the Report.
2. WHEN the User's Streak is broken (no interview completed in a 24-hour period), THE Platform SHALL send an in-app notification encouraging the User to resume practice.
3. WHERE a User has enabled email notifications in their settings, THE Platform SHALL send an email notification for each event described in Criteria 1 and 2.
4. WHEN a User marks a notification as read, THE Platform SHALL update the notification status to "read" and decrement the unread notification count in the UI.

### Requirement 13: Subscription Plans and Payments

**User Story:** As a User, I want to subscribe to a paid plan, so that I can access unlimited interviews and premium features.

#### Acceptance Criteria

1. THE Platform SHALL support at minimum two Plans: "free" (limited to 5 interviews per 24 hours, no PDF export) and "pro" (unlimited interviews, PDF export, shareable report links).
2. WHEN a User initiates a plan upgrade, THE Platform SHALL redirect to a payment provider checkout page and, upon successful payment confirmation, update the User's Plan to "pro" immediately.
3. WHEN a payment is confirmed via webhook, THE Platform SHALL create a Payment record and activate the corresponding Plan features within 60 seconds of receiving the webhook.
4. IF a payment webhook cannot be verified (invalid signature), THEN THE Platform SHALL return a 400 error, log the attempt, and not update the User's Plan.
5. WHEN a User's Plan subscription expires, THE Platform SHALL downgrade the User to the "free" Plan and send an in-app notification informing the User of the change.

---

### Requirement 14: Security and Authorization

**User Story:** As a platform operator, I want all API endpoints and real-time events to be properly authenticated and authorized, so that User data remains private and the system is protected from abuse.

#### Acceptance Criteria

1. THE Platform SHALL require a valid Access_Token in the Authorization header for all non-public API endpoints; requests without a valid token SHALL receive a 401 Unauthorized response.
2. THE Platform SHALL enforce that Users can only read and modify their own data; cross-user access attempts SHALL receive a 403 Forbidden response.
3. WHEN an Access_Token expires, THE Platform SHALL automatically attempt to refresh it using the stored Refresh_Token without interrupting the User's session.
4. THE Auth_Service SHALL store all passwords exclusively as bcrypt hashes; plaintext passwords SHALL never be persisted or logged.
5. THE Platform SHALL apply rate limiting of 100 requests per minute per IP address on authentication endpoints; requests exceeding this limit SHALL receive a 429 Too Many Requests response.
6. THE Platform SHALL sanitize all User-supplied input before persisting to the database to prevent SQL injection and XSS attacks.
7. WHEN a Socket_Server connection is established, THE Socket_Server SHALL verify the Access_Token provided in the connection handshake; connections without a valid token SHALL be rejected immediately.

---

### Requirement 15: Performance and Availability

**User Story:** As a User, I want the platform to respond quickly and remain available, so that my interview practice is not disrupted by slowness or downtime.

#### Acceptance Criteria

1. THE Platform SHALL respond to all non-AI-dependent API requests within 500 milliseconds at the 95th percentile under normal load (up to 100 concurrent users).
2. THE Platform SHALL respond to Dashboard and History page data requests within 1 second at the 95th percentile.
3. WHEN the backend encounters an unhandled exception, THE Platform SHALL log the full stack trace, return a 500 Internal Server Error to the client, and continue processing subsequent requests.
4. THE Platform SHALL serve the frontend initial load (Largest Contentful Paint) within 3 seconds on a standard broadband connection.
5. IF a third-party service (AI_Engine, Speech_Service, payment provider) is unavailable, THEN THE Platform SHALL surface a user-facing error message and fail gracefully without causing a full application crash.
